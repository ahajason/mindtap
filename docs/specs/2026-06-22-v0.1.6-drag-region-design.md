### L3 具体问题(user-facing interaction 全枚举)

> 用户报:"拖动不了"。下面穷举所有 user-facing interaction,每个回答"改后表现"。

| 交互 | 改前表现 | 改后表现(本 spec) |
|---|---|---|
| 拖 Sidebar 顶部 padding(traffic light 旁) | 不能拖 | **能拖** — StyleGuideLayout 独立 drag strip 覆盖 (0,0)~(252,40) |
| 拖 Sidebar header(h1 文字 / padding) | 不能拖(V0.1.4 时代) | **能拖** — aside `data-tauri-drag-region="deep"` |
| 拖 Sidebar 底部 / 左右空白 | 不能拖 | **能拖** — aside deep 覆盖整个 box |
| 拖 Sidebar 内的空白 separator 区域 | 不能拖 | **能拖** — aside deep |
| 点击 Sidebar nav link | 可点 | **仍可点** — nav `data-tauri-drag-region="false"` 显式阻断 drag |
| 拖 Main 顶部 36px 横条 | 不能拖 | **能拖** — main 内部 drag handle div |
| 拖 Main 内容区(空白) | 不能拖 | **不能拖**(符合 V0.1.4 plan 决策:不回到"整窗可拖") |
| 点击 Main 内容 button / link / input | 可点 | **仍可点** — 不挂 drag region 默认 no-drag |
| 双击 Sidebar header | 拖动最大化(NSWindow 行为)? | **不最大化** — drag.js macOS 双击逻辑需 mouseup 时位置不变才触发,toggle_maximize 行为保留 |
| 拖窗口右下角 resize handle | 工作 | **仍工作** — 系统级 resize 与 webview drag region 无关 |
| 拖窗口左边 / 上边 / 右边非 drag 区 | 不能拖 | **不能拖**(符合设计) |
| macOS native traffic light 圆圈 | macOS 系统级,最高优先级 | **仍 macOS 系统级** — drag strip 不覆盖 macOS 层 |
| 文字选(text selection) | 被 NSWindow 抑制(V0.1.1 retro) | **不被抑制** — web drag region 不拦截 mousedown |
| 鼠标光标在 drag region 上 | (V0.1.6 之前) `cursor: grab` | **普通箭头** — macOS HIG 共识:drag 保持原状 |
| 鼠标光标在 resize handle | 系统 resize cursor | **不变** — 系统级 |
| 窗口失焦时 drag region | 可拖 | **仍可拖** — drag region 不依赖 active 状态 |

## 设计原则(后续 routing 页面继承)

1. **每个路由页面 main 区域自带 drag handle** — 顶部 ~36px 横条是用户视觉锚点
2. **drag region 用 `deep` 模式** — 因为 h1 / icon 等子元素常出现
3. **可交互子元素必显式 `data-tauri-drag-region="false"`** — 防止被父 deep 吞掉
4. **不让 traffic light 让位区无 drag** — 必须有独立 strip / handle 覆盖
5. **不写 `cursor: grab/grabbing`** — macOS drag 保持原状是 HIG 共识
6. **capability `core:window:allow-start-dragging` 必须显式列** — 即使 `core:default` 在某些版本包含,显式声明是规范

## 不做

- 不加 `cursor: grab/grabbing`(反 macOS HIG)
- 不退回 NSWindow.setMovableByWindowBackground(V0.1.1 retro 已否决)
- 不给所有 clickable 元素加 `data-tauri-no-drag`(V0.1.4 plan 已否决,工作量 + 易漏)
- 不让 main 内容区可拖(只 drag handle 区)
