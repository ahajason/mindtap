## 1. 背景与目标

### 1.1 现状（V1.2 痛点）

V1.2 将 Mindtap 实现为标准 Mac + Windows 桌面应用（主窗 + 侧栏 + 卡片列表 + 热力图）。多轮用户测试暴露 3 个核心问题：

1. **像"应用程序后台"而非轻量 app**：打开主窗才看到记录、关闭主窗即丢失上下文、缺少常驻存在感
2. **多维度记录操作繁琐**：每次记录需打开主窗 → 选类型 → 选截止 → 写内容 → 保存，至少 5 步；ENFP 用户灵感瞬间流失
3. **TODO 状态机缺失**：V1.2 只有"完成/未开始"二态，无法表达"正在做"的中间状态；用户工作时易忘记正在进行的任务

### 1.2 战略转向（V1.3）

**V1.3 重新定位为"以浮动窗口为核心交互载体"的轻量记录 app**。核心战略：

- **浮动窗常驻**：类似 macOS 灵动岛 / Windows 桌面便签，常驻任意位置
- **主窗降级**：从主要交互面降级为"管理 / 统计 / 历史"次要界面
- **状态机升级**：TODO 从 2 态扩展为 4 态（未开始 / 进行中 / 暂停 / 完成）

### 1.3 目标指标

| 指标 | 目标 | 验证方式 |
|------|------|---------|
| 记录耗时 | < 3 秒 | 用户测试 + 计时器 |
| 查看耗时 | < 1 秒 | 一眼可见浮动窗 |
| 思考成本 | 0 | 无 Tab 切换 / 无类型选择 / 无截止时间强制 |
| 跨平台 | Mac + Windows 原生浮动窗 | 模拟器 + 真机测试 |
| 灵感留存率 | > 80% | 用户自评（10 人小组，1 周追踪） |

### 1.4 不做（Out of Scope）

- 移动端（V2.0 后续）
- 云同步 / 多人协作
- 数据导入导出（V1.4 后续）
- 通知中心 / 提醒功能（V1.4 后续）
- 灵动岛 / Windows 11 小组件（V1.x 跟进）

## 2. 用户画像与设计铁律

### 2.1 目标用户

**ENFP 创意型人格** — 点子爆发但极易遗忘、难以长期坚持记录：

- 同时处理 3-5 个并行任务，频繁切换
- 灵感瞬间即逝，必须 3 秒内记录
- 抗拒复杂操作和填表
- 需要"轻推"（视觉提示）才能想起当前在做什么

### 2.2 唯一设计准则（铁律）

> **3 秒记录、1 秒查看、0 思考成本**

| 维度 | 具体含义 | 约束 |
|------|---------|------|
| 3 秒记录 | 从打开浮动窗到保存记录，< 3 秒完成 | 全局快捷键 ≤ 1 次点击 + 输入 ≤ 10 字 + 保存 1 键 |
| 1 秒查看 | 任意时刻看浮动窗，1 秒内获取信息 | 折叠态 ≤ 36px，常驻桌面任意位置 |
| 0 思考成本 | 无 Tab 切换、无类型选择、无截止时间强制 | 文字按钮 + 默认值 + 智能推断 |

### 2.3 基础特性（V1.0 → V1.3 沿用）

- **无登录注册**：单机使用
- **全量数据本地存储**：SQLite
- **重记录轻查看**：录入优先，统计次要
- **无冗余功能**：砍掉一切不能直接服务记录的功能

## 4. 数据模型

### 4.1 3 源表 + 1 通用 record 表

**核心决策**：拆分 3 源表（语义独立）+ 1 通用 record 表（事件双写，用于主窗统一列表展示）。

| 表 | 角色 | 状态 |
|----|------|------|
| `task` | 待办（4 状态机） | 核心 |
| `idea` | 灵感（极简） | 极简 |
| `check_in` | 打卡（频次） | 频次 |
| `record` | 通用（双写） | 视图层 |

### 4.2 task 表（V1.3 核心）

```sql
CREATE TABLE task (
  id                  INTEGER PRIMARY KEY AUTOINCREMENT,
  content             TEXT    NOT NULL,
  status              TEXT    NOT NULL DEFAULT 'pending',
  -- 'pending' | 'active' | 'paused' | 'done'
  duration_ms         INTEGER NOT NULL DEFAULT 0,
  first_focused_at    INTEGER,
  focus_started_at    INTEGER,
  paused_at           INTEGER,
  focused_count       INTEGER NOT NULL DEFAULT 0,
  due_at              INTEGER,
  created_at          INTEGER NOT NULL,
  updated_at          INTEGER NOT NULL,
  archived_at         INTEGER
);
CREATE INDEX idx_task_status ON task(status);
CREATE INDEX idx_task_due_at ON task(due_at);
```

**4 时间字段语义**（区分"跨度"与"计时"）：

| 字段 | 语义 | 何时变 |
|------|------|--------|
| `first_focused_at` | 首次激活起点（用于"任务总跨度"） | pending → active 时设，后续不变 |
| `focus_started_at` | 本次激活起点（暂停时清零） | 每次 active 重新设；暂停时清 |
| `paused_at` | 本次暂停起点 | active → paused 时设 |
| `duration_ms` | 累计计时（暂停/完成时累加） | 暂停/完成时累加 now - focus_started_at |
| `focused_count` | 累计激活次数 | 每次 pending/paused → active 时 +1 |

### 4.3 idea 表

```sql
CREATE TABLE idea (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  content     TEXT    NOT NULL,
  created_at  INTEGER NOT NULL
);
```

### 4.4 check_in 表

```sql
CREATE TABLE check_in (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  habit       TEXT    NOT NULL,
  note        TEXT,
  checked_at  INTEGER NOT NULL
);
```

### 4.5 record 表（通用，双写）

```sql
CREATE TABLE record (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  kind        TEXT    NOT NULL,        -- 'task' | 'idea' | 'check_in'
  source_id   INTEGER NOT NULL,        -- 指向 task/idea/check_in.id
  content     TEXT    NOT NULL,
  status      TEXT,                    -- task 才有；idea/check_in NULL
  created_at  INTEGER NOT NULL
);
CREATE INDEX idx_record_created_at ON record(created_at);
CREATE INDEX idx_record_kind ON record(kind);
CREATE INDEX idx_record_status ON record(status);
```

### 4.6 业务层双写（D4-B 锁定）

**核心原则**：源表 INSERT + record INSERT 必须同事务，**全部成功或全部回滚**。

**Rust 业务函数约束**：
- 所有双写函数必须用 `conn.transaction()` 包裹
- `tx.last_insert_rowid()` 紧跟 INSERT 取 source_id
- 业务层禁止直接 SQL，必须走 `create_*` / `update_*` / `delete_*`
- 单元测试必须覆盖每个双写函数

```rust
// 示例：create_task
pub fn create_task(conn: &mut Connection, content: &str,
                   due_at: Option<i64>) -> rusqlite::Result<Task> {
    let now = now_ms();
    let tx = conn.transaction()?;
    tx.execute("INSERT INTO task (content, status, created_at, updated_at, due_at)
                VALUES (?1, 'pending', ?2, ?2, ?3)",
               params![content, now, due_at])?;
    let task_id = tx.last_insert_rowid();
    tx.execute("INSERT INTO record (kind, source_id, content, status, created_at)
                VALUES ('task', ?1, ?2, 'pending', ?3)",
               params![task_id, content, now])?;
    tx.commit()?;
    Ok(Task { id: task_id, ... })
}
```

**同类函数**：`update_task_status`（同步 record.status）、`delete_task`（先删 record 再删源）、`create_idea`、`create_check_in`。

## 5. 4 状态 TODO 状态机

### 5.1 状态机

```
                     ┌──────────┐
                     │ pending  │  ← 创建
                     └────┬─────┘
                          │ ⏵ 开始计时
                          ▼
                     ┌──────────┐
                     │  active  │  ← Focus 块唯一
                     └────┬─────┘
                          │
              ┌───────────┴───────────┐
              ▼                       ▼
         ┌────────┐              ┌────────┐
         │ paused │              │  done  │  ← Focus 块变体
         └───┬────┘              └───┬────┘
             │ ⏸ 继续                │ ↶ 撤销 / 新任务覆盖归档
             │                       │
             ▼                       ▼
         ┌────────┐              归档
         │ active │
         └────────┘
```

### 5.2 状态不变量（写操作必须满足）

| # | 不变量 | 说明 |
|---|--------|------|
| L1 | 全局唯一 `status='active'` | 同时只能有一个 Focus |
| L2 | `paused → active` 时 | `focus_started_at = now()`，`duration_ms` 保留 |
| L3 | `active → paused` 时 | `duration_ms += now - focus_started_at`，`focus_started_at = null`，`paused_at = now` |
| L4 | `active → done` 时 | 累加 `duration_ms`，进入 done 态 |
| L5 | `done → *` 不允许 | done 终态；可"撤销"回 active（重置 `focus_started_at`，`duration_ms` 保留） |

### 5.3 状态 vs 视觉

| 状态 | 标签颜色 | 主 Focus 块 | ⇄ 切换下拉 |
|------|---------|-----------|-----------|
| pending | 灰 `rgba(0,0,0,0.05)` | — | ✓ 列出 |
| **active** | 绿 `rgba(52,199,89,0.18)` | ✓ 唯一 | — |
| paused | 橙 `rgba(255,149,0,0.18)` | — | ✓ 列出（带已计时） |
| done | 淡绿 `rgba(52,199,89,0.12)` | ✓ 变体 | — |

### 5.4 关键流程

| 操作 | 状态变化 | Focus 块 |
|------|---------|---------|
| 点 ✓ 完成 | active → done | active 态 → done 态（同任务，时间冻结 50% 透明） |
| 点 ↶ 撤销 | done → active | done 态 → active 态（呼吸灯亮起，时间从 now 开始） |
| 点 ⏵ 开始计时（新任务） | 新 task → active，旧 done 自动 archived | 旧 done 被覆盖 |
| 点 ⇄ 切换 | 当前 active → pending/paused，新 id → active | 旧 active 被覆盖 |

**done 态保留规则**：直到被新任务 ⏵ 主动覆盖 / 主动 ⇄ 切换。归档（archived_at 非空）后从主窗时间线隐藏，独立"归档" tab 可见。

## 7. 核心用户旅程

### 7.1 3 秒记录

```
T+0.0s   按 Cmd+Shift+Space
T+0.1s   浮动窗展开（折叠态 36px → 展开态 280px，200ms 动画）
T+0.5s   用户输入文字 "回复张三关于 Q3 的邮件"
T+1.5s   点 ⏵ 开始计时
T+1.6s   写 DB（task + record 事务双写，task.status='active'）
         浮动窗折叠（280px → 36px，200ms 动画）
T+1.8s   主 Focus 块显示该任务 + 时间从 00:00:00 开始走
T+2.0s   用户可继续工作

✓ 总耗时 < 3s
```

### 7.2 1 秒查看

浮动窗常驻桌面任意位置

     ↑ 呼吸灯   ↑ 标签   ↑ 标题      ↑ 计时  ↑ 状态

### 7.3 切换 Focus

```
1. 浮动窗展开
2. 点 ⇄ ▼ 顶靠右下拉
3. 下拉显示：未开始（灰）+ 已暂停（橙）清单
4. 点目标项右侧 ⏵
5. 后端 task_switch 原子操作：
   - 当前 active → pending/paused
   - 新 id → active (focus_started_at = now)
6. 浮动窗主 Focus 块自动更新（evt_focus_changed）
7. 下拉自动关闭，浮动窗回到默认展开态
```

### 7.4 暂停 vs 完成

| 操作 | 状态变化 | 累计 | 列表 | 视觉 |
|------|---------|------|------|------|
| `⏸ 暂停` | active → paused | `duration_ms += now - focus_started_at` | 出现在切换下拉 | 暂停按钮浅蓝（突出） |
| `✓ 完成` | active → done | `duration_ms` 累加完毕 | 从切换下拉消失 | 完成按钮淡绿（弱） |

### 7.5 截止快捷

- `📅 稍后` → `due_at = end_of_today(今天 23:59)`
- `📅 明天` → `due_at = end_of_tomorrow(明天 23:59)`
- 仅设置 `due_at`，不影响 status（仍要点 ⏵ 开始计时）
- 主窗列表按 due_at 排序

| 维度 | macOS | Windows |
|------|-------|---------|

| 模糊性能 | Metal 加速 | DWM 加速（Win 10/11） |

### 9.2 关键技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 跨平台框架 | Tauri 2 | Web 前端复用 + Rust 后端 + 小体积 |
| 状态管理 | Zustand / Jotai | 轻量 + 适合悬浮窗 |
| 时间显示 | 后端每秒 evt_tick | 前端不计时，状态权威 |
| 数据库 | SQLite (rusqlite) | 单机本地 |
| 迁移工具 | 不需要（V1.0 未发布） | — |
| 测试 | Vitest（前端）+ cargo test（Rust） | 双层 |

### 9.3 未决项
