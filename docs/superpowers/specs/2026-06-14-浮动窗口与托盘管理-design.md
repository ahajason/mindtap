# Mindtap V1.4 浮动窗口与托盘管理 · 设计文档

> **状态**：设计中（待用户审阅）
> **创建**：2026-06-14
> **作者**：Jason + Claude
> **关联**：V1.3 浮动窗口快速记录（已发布，git 780072b）的生命周期补丁；本设计不引入新交互，只补齐「窗口管理」与「系统托盘」两条遗失的用户路径

---

## 1. 背景与目标

### 1.1 现状（V1.3 痛点）

V1.3 实现了浮动窗快速记录：浮窗 320×36 折叠 / 360×280 展开、`Cmd/Ctrl+Shift+Space` 全局 toggle、FAB 主窗入口、4 状态任务机。多轮真机测试暴露 3 个管理面问题：

1. **浮窗无法被关掉**：折叠 / 展开态都没有 `×`，原生 decorations=false，按下全局快捷键只是 toggle。要"暂时摆脱"浮窗只能等待它正好处于显示态再 toggle 一次。
2. **Windows / Linux 上浮窗失踪**：浮窗 `skipTaskbar=true` + `decorations=false` + 任务栏隐藏意味着用户**没有任何视觉锚点**找回它。快捷键能 toggle，但用户常常不记得快捷键。
3. **主窗 `X` = 退应用**：V1.3 主窗有默认 decorations，`X` 直接 `app.exit()`。用户体验割裂：主窗和浮窗是同一应用的两个面，关其中一个不应该杀进程。

### 1.2 V1.4 战略定位

**V1.4 不引入新交互，只补齐 V1.3 漏掉的生命周期管理路径**：

- **窗口关闭 = 隐藏到托盘**（与 Raycast / 滴答清单等"工具型 APP"惯例一致）
- **系统托盘提供唯一可寻址的"应用根"**（macOS 菜单栏 / Windows 系统托盘 / Linux 托盘）
- **退出必须显式**（避免误操作杀进程）

### 1.3 目标指标

| 指标 | 目标 | 验证方式 |
|---|---|---|
| 浮窗唤起路径数 | ≥ 4（快捷键 / FAB / 侧栏 / 托盘） | 手工 E2E |
| 窗口 `X` 行为一致 | 主窗 + 浮窗 = 隐藏，进程不退 | 手工 E2E |
| 退出误操作率 | = 0 | 确认弹窗拦截 |
| 跨平台一致 | macOS / Windows / Linux 均提供托盘入口 | 三平台真机 |
| 浮窗唤起延迟 | < 200ms（hide→show 无重建） | stopwatch |

### 1.4 不做（Out of Scope）

- 设置面板（autostart / 全局快捷键修改都通过托盘菜单）
- 浮窗拖动吸附、屏幕边缘智能停靠
- 通知中心 / 提醒 / 横幅
- 多实例 / 单实例锁（V1.4 不处理"双启动"）
- 自定义托盘图标主题

## 2. 设计铁律（沿用 V1.3）

> **3 秒记录 / 1 秒查看 / 0 思考成本**

| 维度 | V1.4 落地 |
|---|---|
| 3 秒记录 | 浮窗始终可被 4 条路径 ≤1 步唤起（不引入"先开设置 → 启用浮窗"两步） |
| 1 秒查看 | 托盘图标始终存在，是用户"应用还活着"的视觉锚点 |
| 0 思考成本 | 关闭 = 隐藏（无"真的要关吗？"弹窗）；退出 = 显式托盘菜单项（仅此处要确认） |

## 3. 关键决策（D1–D8）

| ID | 决策 | 理由 |
|---|---|---|
| D1 | 窗口 `X` = `prevent_close() + hide()`，不进 IPC 也不弹窗 | 与"3 秒记录"零成本原则一致；不引入新概念 |
| D2 | 主窗与浮窗统一应用 `prevent_close` 拦截 | 两窗口都是同一应用的入口，行为必须一致 |
| D3 | 系统托盘必须三平台都注册 | 用户视角"应用有根"，跨平台体验一致 |
| D4 | 托盘左键 = toggle 浮窗；右键 = 菜单 | 左键为最高频操作（找回浮窗）让路 |
| D5 | 托盘菜单 4 项：toggle 浮窗 / toggle 主窗 / autostart checkbox / 退出（带确认） | 覆盖所有窗口管理需求，不做设置页 |
| D6 | 退出 = 托盘菜单专属入口 + 二次确认弹窗 | 唯一显式退出路径；防止误关 |
| D7 | 不解析 `--floating` 启动参数；任何启动都默认浮窗可见 | 与"V1.3 体验延续"原则一致 |
| D8 | 不引入新设置面板；所有配置走托盘菜单 | YAGNI；遵循"无冗余功能"项目铁律 |

## 5. 托盘菜单与点击行为

### 5.1 菜单结构

```
┌──────────────────────────────┐
│ 显示/隐藏 浮窗      ⌘⇧Space  │   MenuItem
│ 显示/隐藏 主窗               │   MenuItem
├──────────────────────────────┤   Separator
│ ☑ 开机自启                   │   CheckMenuItem（启动时 sync）
├──────────────────────────────┤   Separator
│ 退出 Mindtap                 │   MenuItem
└──────────────────────────────┘
```

### 5.2 点击路由

| 来源 | 平台 | 行为 |
|---|---|---|
| 左键托盘 | 全平台 | `window_toggle("floating")` |
| 右键托盘 | 全平台 | 系统默认弹出菜单 |
| 菜单项「显示/隐藏 浮窗」 | 全平台 | `window_toggle("floating")` |
| 菜单项「显示/隐藏 主窗」 | 全平台 | `window_toggle("main")` |
| 菜单项「开机自启」 | 全平台 | `autostart_set(!current)`，立即重读真实状态并 sync 勾选 |
| 菜单项「退出 Mindtap」 | 全平台 | `quit_with_confirm()` → 确认后 `app.exit(0)` |

### 5.3 菜单状态保持

- 启动时 `setup` 阶段调 `sync_menu_check()` 读 autostart 实际状态写入 CheckMenuItem
- 用户点击 checkbox 后：先调 `set_enabled` 写实际状态，再 `sync_menu_check` 校验（防漂移）
- 不持久化到 SQLite —— autostart 状态由系统级（macOS LaunchAgent / Windows Registry）保管

### 5.4 图标

- 复用 `src-tauri/icons/32x32.png` 作为托盘底图
- macOS：调用 `image.as_template(true)` 让系统接管黑白
- Windows：普通 PNG，alpha 通道保留
- Linux：PNG（部分 DE 需 SVG，但 V1.4 接受"某些 DE 显示欠佳"）

## 6. 窗口生命周期

### 6.1 拦截 CloseRequested

```rust
// 在 lib::setup 里
for label in ["main", "floating"] {
    if let Some(w) = app.get_webview_window(label) {
        let w_clone = w.clone();
        w.on_window_event(move |event| {
            if let WindowEvent::CloseRequested { api, .. } = event {
                api.prevent_close();
                let _ = w_clone.hide();
            }
        });
    }
}
```

### 6.2 关键性质

- **hide 状态保留 webview 进程**：`show()` 瞬间恢复（实测 < 50ms）
- **不破坏 V1.3 全局快捷键 toggle**：`floating_toggle` / `window_toggle("floating")` 调 `is_visible()` 后 `show() / hide()`，无重建开销
- **`tauri.conf.json` 启动配置不变**：`visible: true`（两窗口都启动可见，与 D7 一致）
- **`skipTaskbar: true` 不变**（浮窗始终不占任务栏）
- **autostart 参数清空**：`tauri_plugin_autostart::init(..., Some(vec!["--floating"]))` 改回 `None`（澄清 #4 不需要此参数）

### 6.3 macOS NSPanel 行为兼容

- 已有 `HidesOnDeactivate: false`：浮窗不会被 deactive 触发自动隐藏
- 浮窗 `decorations: false` 本就没有红绿灯 → `X` 拦截对浮窗无视觉影响（用户根本看不到关闭按钮）
- 主窗有原生红绿灯 → 拦截后视觉上"点击红绿灯 = 隐藏到托盘"

## 7. 退出确认

### 7.1 触发条件

| 入口 | 是否确认 |
|---|---|
| 托盘菜单「退出 Mindtap」 | ✅ 确认 |
| 窗口 `X` | ❌ 不涉及（已拦截为 hide） |
| 全局快捷键 | ❌ 不涉及 |
| 任何其他入口 | ❌ V1.4 不提供其他入口 |

### 7.2 弹窗规格

- 库：`tauri-plugin-dialog::DialogExt::message` + `blocking_show`
- macOS：sheet 形式（`parent: Some("main")`）
- Windows：模态任务对话框
- 文案：
  - **标题**：退出 Mindtap
  - **内容**：退出后，浮窗与主窗都会关闭。下次需要时可从 Dock / 任务栏图标重新打开。
  - **按钮**：退出（destructive 样式）+ 取消（默认焦点）

### 7.3 实现位置：`src-tauri/src/tray/confirm.rs`

```rust
pub async fn ask_quit_confirm(app: &AppHandle) -> bool {
    use tauri_plugin_dialog::{DialogExt, MessageDialogButtons, MessageDialogKind};
    app.dialog()
        .message("退出后，浮窗与主窗都会关闭。\n下次需要时可从 Dock / 任务栏图标重新打开。")
        .title("退出 Mindtap")
        .kind(MessageDialogKind::Warning)
        .buttons(MessageDialogButtons::OkCancelCustom(
            "退出".to_string(), "取消".to_string()
        ))
        .blocking_show()
}
```

### 7.4 不做

- ❌ 拦截 Cmd+Q / Alt+F4 二次确认（YAGNI；macOS 用户依赖 Cmd+Q 强退）
- ❌ "退出后保留托盘图标"等扩展模式

## 9. 错误处理

| 错误源 | 处理 | 用户感知 |
|---|---|---|
| `app.get_webview_window(label)` 返回 None | `eprintln!` + IPC 返回 `Result::Err` | 前端 console 报错 |
| `tray_icon_builder.build()` 失败 | setup 阶段返回 `Err`，启动失败 | 开发者可见 panic 日志 |
| `app.dialog().blocking_show()` 失败 | 视为"用户取消"（保守默认） | 静默不退出 |
| `autostart_set` 失败 | CheckMenuItem 恢复旧值 + `eprintln!` | 菜单勾选闪烁回滚 |
| `global_shortcut.register()` 失败 | 启动失败 | 启动失败 |
| `app.exit(0)` 失败（基本不会） | `let _ = ...;` 忽略 | 静默 |

**统一约定**：
- IPC 函数统一 `Result<T, String>`，错误信息中文化
- 前端不主动 try/catch（让 React StrictMode 下的未处理 rejection 可见，沿用 V1.3 模式）
- 不引入 `thiserror` 自定义枚举（保持 `String` 错误简单）

**不做**：Sentry / Crashpad 上报；静默 fallback。

## 11. 风险与回滚

| 风险 | 缓解 |
|---|---|
| `prevent_close` 在某些 Tauri 2 版本有 bug | 升级到最新 2.x；如无法修复则退回到 "X = app.exit()" 并接受 V1.3 现状 |
| `tauri-plugin-tray-icon` 在 Linux Wayland 兼容性差 | 文档标注"Wayland 体验降级"；不阻塞其他平台发布 |
| 用户报告 "X 关不掉 app 不习惯" | 在 README 加 "Mindtap 是常驻工具型 APP" 说明 |
| autostart 状态在系统层与菜单层漂移 | 每次菜单点击后 `sync_menu_check` 强制对齐 |
