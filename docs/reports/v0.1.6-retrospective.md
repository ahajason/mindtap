# V0.1.6 Retrospective — Drag 修复走了 5 次才到位(基于三层决策法沉淀)

> 创建: 2026-06-22
> 触发: V0.1.6 drag region 修复,从 plan 到可工作改了 5 次;用户追问"为什么改了这么多次,前面为什么都没发现"
> 设计沉淀: `docs/specs/2026-06-22-v0.1.6-drag-region-design.md`
> 平台常识: `_project/macos-drag-cursor-design.md`

## 1. Initial decision

### V0.1.4 commit `20e9ac9` 时的目标

- **目标**: 修 V0.1.1 retro 暴露的 3 个 NSWindow 副作用(文字选不中 / resize 失效 / cursor 不变)
- **方案**: NSWindow.setMovableByWindowBackground(true) → web 侧 `data-tauri-drag-region`
- **verification scope**: "V0.1.1 那 3 症状消失" = 修复成功信号

### V0.1.6 plan 时的目标(我接到用户反馈"拖不动")

- **目标**: 缩 drag region 到 Sidebar header + Main 顶部 36px,避免 button 阻断
- **方案**: 改 StyleGuideLayout + Sidebar + 加 CSS `-webkit-app-region: drag`
- **verification scope**: "代码 + 单测 + tsc + cargo 通过" = 修复成功(**没把 dev 实测拖动列进去**)

## 2. Unidentified at decision time

### L1(原始权威)漏查 — 每次修复都漏

| 阶段 | 漏的 L1 |
|---|---|
| V0.1.4 | Tauri 2 `data-tauri-drag-region` 文档没读全。**文档实际说: macOS 走 JS IPC(自动注入 drag.js),Windows 走 `app-region: drag` CSS**。commit 切方案时把两个路径都漏了,既没配 CSS 也没开权限 |
| V0.1.6 plan | webfetch 了文档,但**只看第一页结论**,没看"Tauri 自己注入 drag.js"那段源码;也没意识到 `app-region: drag` 是 Windows WebView2 123+ 触控专属 |
| V0.1.6 加 webkit-app-region CSS 时 | macOS WKWebView 实际**忽略**这个 CSS(Tauri 不读)。误以为是 macOS 兼容写法,实际是死代码 |
| V0.1.6 自加 `cursor: grab/grabbing` 时 | 完全没查 macOS HIG,自作主张加视觉反馈 — **反 HIG 共识**,macOS 上 drag 应保持普通箭头 |

### L2(项目设计)漏查 — 致命点

| 阶段 | 漏的 L2 |
|---|---|
| V0.1.4 | **没把"dev 环境实测拖动"作为 acceptance criteria**。commit message 写"切方案"就算收口 |
| V0.1.6 plan(我) | **没问用户"V0.1.4 当时有没有实测过拖动"**。如果当时就没实测,V0.1.4 修复从来就没生效过 — 这是关键起点确认 |
| V0.1.6 自加 cursor CSS 时 | **没查项目 design-system spec**(`glassmorphism-impl-spec.md`),该 spec 也没规定 drag region cursor 行为。漏的 L2 项目规范 |
| V0.1.6 capability | **没看现有 capability 文件**。如果当时看了 default.json,会发现只有 `core:default`,显式列 `core:window:allow-start-dragging` 是必需补充项 |

### L3(具体问题)漏查 — 真凶

| 阶段 | 漏的 L3 |
|---|---|
| V0.1.4 | **"症状消失" ≠ "修复生效"**。V0.1.1 那 3 症状"消失"不是因为 web drag 替代了 NSWindow + 没副作用,而是因为 web drag **根本没生效**,NSWindow 副作用 + web drag 副作用**一起**没了(因为根本没 drag) |
| V0.1.6 plan | **没问"现在拖时光标变不变"**就直接推断根因。但即使问了,**macOS 上光标也不会变**(HIG 设计),所以光标分诊**本身就是错的诊断信号** |
| V0.1.6 修 CSS | **没在 commit 前实测**。3 个修复都是"代码改完 + 单测过 = 完成",没有 "用户桌面拖一次" 步骤 |
| V0.1.6 drag region 边界 | 没枚举所有 drag region 应当工作的区域:traffic light 让位区(Sidebar margin 父 div 区域) / Sidebar header(h1 子元素阻断) / Sidebar padding(没被 header 覆盖) / Sidebar nav(clickable 需 false) — 每次都是用户实测反馈才发现下一个边界 |

## 3. Later exposure

### 真根因链(V0.1.6 实测逐步暴露)

| 用户反馈 | 我推断 | 真根因 |
|---|---|---|
| "拖不动" | V0.1.4 漏 L3 反向副作用(button 自动 reset) | **V0.1.4 修复从来没生效过**(缺 CSS + 缺权限) |
| "光标 grab 但不能拖" | 我加 `-webkit-app-region: drag` 应能修 | macOS Tauri 不读 CSS,走 JS IPC;**真正缺的是 `core:window:allow-start-dragging` 权限** |
| "main 顶部能拖,sidebar header 不能" | Sidebar header 需要 `deep` 模式 | drag.js 逻辑:`attr === '' → el === composedPath[0]`。header 含 h1 子元素,点 h1 时 composedPath[0] 不是 header |
| "sidebar padding 不能拖" | aside 整体加 deep | drag region 覆盖了,但**mt-10 让位区在父 div 内,不在 aside 内** |
| "sidebar padding 还是不能拖" | 加 drag strip | aside margin 区 = 父 div 区域,确实需要独立 drag handle 覆盖 |
| (用户指出)"macOS drag 光标不该变" | 我推断用光标当诊断信号 | **macOS HIG 共识: drag 时光标保持普通箭头,光标不是 drag 状态机**。我之前写的 `cursor: grab/grabbing` 反 HIG |

## 4. Generalized lesson

**核心 lesson:**

> **改 system API 时,"反向症状消失"不是修复成功的证据**。当修复是反向/替代性的(切技术栈),反向症状消失有两种解释:
> 1. 修复生效,新机制替代旧机制且无副作用 ✓
> 2. 新机制根本没运行,旧机制被下线 = 双方都没在跑 ✗
>
> **区分两者的唯一可靠手段是 dev 环境实测新机制本身在工作**(不能只看反向症状是否消失)。这个必须成为 system API 改动的 hard acceptance criteria,不是 nice-to-have。

**衍生 lesson 1 — 诊断信号 vs 装饰信号:**

> macOS 拖动光标是 AppKit 系统行为,跟 webview drag region 是否工作**完全无关**。不要把光标变不变当诊断信号。**唯一可靠的 drag 验证 = 实际拖动窗口看位置变化**。

**衍生 lesson 2 — pre-clarification 不该用错信号:**

> 用户报"X 不工作"时,优先问**系统反应**(窗口位置 / 日志 / 控制台错误),不要先问**视觉装饰**(光标 / 动画)。视觉装饰受平台设计约束,常被误读;系统反应是 ground truth。

**衍生 lesson 3 — 反 HIG 的自加行为:**

> V0.1.6 我自作主张加了 `cursor: grab/grabbing`,没查 macOS HIG。这是违反 [[strict-literal-execute]] 规则的"指令外操作":用户没要求,我没问就加,且违反平台规范。**任何 platform-specific 自加行为,先查 HIG/官方规范,再决定加不加**。

**衍生 lesson 4 — L1 文档完整性:**

> 改 system API 前 WebFetch 官方文档时,**必须读完整**(特别是"limitations" / "platform notes" / "implementation details"小节),不能只看首页结论。Tauri 2 文档首页说"用 data-tauri-drag-region",但没提 macOS 走 JS IPC、Windows 走 CSS 的平台差异 — 这个差异在文档别处。

**衍生 lesson 5 — 调研不只是 web search,要看项目内部历史(本对话第二次复盘发现)**

> V0.1.6 我接到"拖不动"反馈时,只看了 V0.1.4 commit diff + 派 subagent webfetch,**完全没看项目内部已有的调研成果**。这些文档全部存在:
>
> | 漏看的内部调研 | 内容 |
> |---|---|
> | `docs/research/.../0-originals/tauri/v2-window-customization.md` | Tauri 官方文档 verbatim 缓存,含警告"data-tauri-drag-region / webkit-app-region: drag 在 transparent window 下不可靠" |
> | `docs/reports/v0.1.1-retrospective.md` | V0.1.1 retro 明确写"修复 = 切到 data-tauri-drag-region"+ smoke test 失败模式 |
> | `docs/archive/v0.1/plans/2026-06-21-meta-methodology-consolidation.md` Step 8.1 / 8.6 | V0.1.4 plan 有 smoke test 步骤(写 + 跑),但 commit 没执行 |
> | `docs/reports/2026-06-21-v0.1.4-fixes.md` | release notes 自己写"待用户交互式验证 4 个修复行为" |
> | V0.1.1 commit `8622a47` 验证段 | "tauri dev 启动,整窗背景按下可拖动" — V0.1.1 当时真的 work |
> | V0.1.4 commit `20e9ac9` | **没有"验证"段** — 对比 V0.1.1 commit 验证段,缺失就是当时漏 |
>
> 调研优先级:**项目内部历史 > web search**。本地缓存的官方文档是已 fetch 成本已付的;旧 retro / 旧 plan 是已思考过的反思;旧 commit 验证段是当时的 dev 实测证据。
>
> **下次接到 bug 的"第一步必做"**:扫 **7** 个内部调研路径(0-originals / 旧 retro / 旧 plan / 旧 release notes / 旧 commit 验证段 / **`.archive/src/` 完整历史 codebase** / `.archive/docs/`)+ 派 1-3 个 subagent 外部 search(**其中 1 个专门扫 `.archive/src/`**),双层并行。
>
> **V0.1.6 我在 retro 时也只是"口头说应该查 `.archive/`",没真正派 subagent 系统扫**。**用户强行要求才补** — 见 lesson 7。

**衍生 lesson 6 — "知道没验证"不等于"待补验证"**

> V0.1.4 release notes `8c97bcf` 自己写 "**待用户交互式验证 4 个修复行为**" — 作者当时知道没 dev 实测,但还是 merge 了。**TODO 不该跟 merge 一起出现**。要么先验证再 merge,要么拆 commit(verify TODO 单独 commit),不能合成一个。
>
> V0.1.4 的合并 commit `5e34493` 把 "TODO: 待用户交互式验证" 当作可 ship 状态 — 这是把 dev 实测从"merge gate"降级为"post-merge follow-up",后果是闷 3 个版本。
>
> **新规则**:任何 commit message / release notes / plan 出现 "待 ... 验证" / "TODO: 验证" / "未实测" 字样,该 commit **不该 merge**。

**衍生 lesson 7 — 规则漏写路径 + 规则没 self-apply 是双重错误(本对话第三次复盘发现)**

> V0.1.6 retro 时我自己写了 `first-step-research` 规则,但**规则本身漏写了 `.archive/src/` 路径** — 只列了 `.archive/docs/`,**没列 `.archive/src/`(完整 V1.0 codebase)**。subagent 系统扫了才发现:
>
> | `.archive/src/` 早就踩过 + 修过的同款问题 | 文件 + 行号 |
> |---|---|
> | macOS drag 走对 IPC: `getCurrentWindow().startDragging()` | `.archive/src/floating/hooks/useDragLongPress.ts:49` |
> | macOS drag cursor 反 HIG 反模式: `cursor-grab active:cursor-grabbing` | `.archive/src/floating/FloatShell.tsx:104` |
> | drag region 边界 deep + nav false 正确模式 | `.archive/src/floating/OuterShell.tsx` |
>
> 也就是说:**浮窗 codebase 早就踩过 + 修过同样问题,只是 style guide 没 grep**。
>
> **错误 1**:我写规则时漏了 `.archive/src/` 这个**最关键**的路径(complete codebase > docs)。规则没覆盖 = 规则无效。
>
> **错误 2**(更严重):retro 时我口头说"应该查 `.archive/`",但**没真正派 subagent 系统扫**,只人工 grep 一两个文件。沉淀的 lesson 5 当时只列 6 项漏看的内部调研,**漏看 `.archive/src/floating/` 的全部内容**。
>
> **用户反馈原文**(本对话第三次复盘触发):
> > ".archive/ 这个目录下的相关内容,你即使是在复盘的时候,也没有真正的去派子 agent 去查询、排查问题"
>
> **双重错误教训**:
> 1. 写规则时必须 self-apply 检验:**所有关键路径都得写进规则**,不能"想到了再补"
> 2. 执行规则时必须真正执行:**派 subagent 系统扫 ≠ 人工 grep 一两个文件**
>
> 沉淀:
> - `first-step-research` 6 路径 → 7 路径(加 `.archive/src/` 完整 V1.0 codebase)
> - 新 `_rules/archive-source-codebase-is-truth.md` — `.archive/src/` 不只是 docs,是已踩坑 + 已修过的 ground truth
> - 新 `_rules/self-apply-after-write.md` — 写规则后必跑 7 项 self-apply 检验(漏路径 / 缺执行步骤 / 没触发条件 = 规则无效)

**衍生 lesson 8 — 沉淀方式不需要用户提示(本对话第三次复盘的核心反馈)**

> 用户原话:"这几个沉淀的方式都应该不需要我提示,你就应该知道。你都没有把这些规则记录到内容里吗?"
>
> 我之前犯的模式:
> 1. 用户反馈 X
> 2. 我沉淀 rule Y
> 3. 下次用户反馈类似 X
> 4. 我**不 apply rule Y**,用户**再提示**才 apply
> 5. 用户问"为什么需要我提示"
>
> 沉淀规则 ≠ 规则生效。规则要 trigger 实际行为改变才算写对。
>
> **下次写完任何 rule/memory 后必跑 7 项 self-apply 检验**:
>
> | # | 检验项 | 不通过时 |
> |---|---|---|
> | 1 | 关键路径列全了吗?(本次漏 `.archive/src/`) | 补路径 |
> | 2 | 有具体执行步骤吗?(不是"应该考虑") | 改 step-by-step |
> | 3 | 有触发条件吗?(什么时候必跑) | 加"何时用"段 |
> | 4 | 有反模式例子吗?(我之前怎么犯的) | 加"反模式"段 |
> | 5 | 跟其他 rule 交叉引用了吗? | 加"## 跟其他规则关系"段 |
> | 6 | 有验证机制吗?(怎么知道 apply 了) | 加 checklist |
> | 7 | 当下能说"下次出现 X,我会自动做 Y"吗? | 如果说不出,规则不够具体,继续改 |

### 一句话总结

> **改 system API 必须是"dev 实测新机制本身在工作"作为硬门槛,反向症状消失不是修复成功的证据;接到"X 不工作"反馈时,先问系统反应不要先问视觉装饰;不要自作主张加违反平台 HIG 的自加行为;调研不只是 web search,要先扫项目内部历史(0-originals + retro + 旧 plan + 旧 release notes + 旧 commit 验证段 + `.archive/src/` 完整 V1.0 codebase + `.archive/docs/`,**其中 `.archive/src/` 是最高优先级**);commit 不能带 "待验证" TODO 一起 merge;规则漏写路径 + 规则没 self-apply 是双重错误,写完规则必跑 7 项 self-apply 检验。**

### 下次类似场景的检查清单

- [ ] 改 system API 前:**先扫项目内部调研**(**7 路径**,含 `.archive/src/`)+ 派 1-3 个 subagent 外部 search(**其中 1 个专门扫 `.archive/src/`**),**双层并行**
- [ ] 改 system API 前:WebFetch 官方文档,**完整看一遍**(包括 limitations / platform notes)
- [ ] 改 system API 前:看官方 runtime 源码(wry/tauri 等),不要只看文档摘要
- [ ] 改 system API 前:用户报"X 不工作"时,先问系统反应(窗口位置 / 日志 / 控制台),不要先问视觉装饰
- [ ] 改 system API 前:**任何 platform-specific 自加行为,先查 HIG/官方规范,再决定加不加**
- [ ] 改 system API 前:**grep `.archive/src/` 找参考实现** — 不只查 docs,要查完整 codebase
- [ ] 改 system API 后:**必须 dev 实测新机制生效**(不是只看反向症状消失)
- [ ] 改 system API 后:`cargo check` / `tsc` / 单测 = 代码没语法错,**不等于**新 API 工作
- [ ] **Commit gate**:commit message 必须含"验证: ..."段(V0.1.1 commit `8622a47` 格式),无验证段 = 不 merge
- [ ] **Commit gate**:commit message 不能带 "待验证" / "TODO: 验证" / "未实测" 字样,有就不 merge
- [ ] **Rule gate**(写完新 rule 后):跑 7 项 self-apply 检验(关键路径 / 执行步骤 / 触发条件 / 反模式 / 交叉引用 / 验证机制 / 当场能复述),不通过 = 规则无效,继续改
