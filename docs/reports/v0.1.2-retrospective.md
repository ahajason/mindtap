# V0.1.2 Retrospective — 玻璃问题修复(P0/P1)

> 创建: 2026-06-21
> 触发: 闭环复盘方法论上线后的首批 retro 验证

## 1. Initial Decision

### 目标

V0.1.1 玻璃感 + 透明窗口交付后,用户反馈仍有问题,需要修到"真玻璃":
- P0-1:body 透明导致桌面窗口文字穿透
- P0-2:spacing token 编号跟 Tailwind 默认错位
- P1:focus state 没有视觉反馈
- P1:window active/inactive 玻璃 vibrance 没区分
- P2:reduced transparency / reduced motion 无降级

### 范围

每个 issue 有独立文件 `docs/research/2026-06-21-v0.1.2-glass-issues/2-issues/<P0|P1|P2>-<topic>/<n>-<name>.md`

### 验证手段

- `tsc --noEmit` 通过
- 视觉 smoke 对比(肉眼)
- 几个新增 unit test

## 2. Unidentified at Decision Time

### L1 原始权威 — 基本穷尽

读了 Apple HIG(Liquid Glass / focus state / accessibility 段)、WCAG AA 4.5:1 标准、MDN `:focus-visible`、`prefers-reduced-*` 媒体查询文档。

### L2 统一设计 — **spec 缺口被显式记录** ⭐ 良好

发现 `docs/design/glassic-ui-spec.md` 当时还在 `.archive/` 里没恢复,V0.1.2 plan 第一步就是 cp 回来 + 对齐 spacing token 编号(spec §四)。

**优点**:spec 缺口被显式识别并修,没绕过。

### L3 具体问题 — **每 issue 都做了用户视角枚举**

每个 P0/P1 issue 文件都有"用户视角表现 + 期望行为 + 实际行为"段。Layer 3 主动枚举做得好。

### 唯一未识别 — body transparent 副作用比预期深

P0-1 issue 描述:body transparent 导致桌面文字穿透。**修复方案选了 body rgba 0.92** 而不是恢复非透明 — 因为要保留玻璃 vibrance。这个选择**没在 issue 文件里记录 trade-off**,直到 V0.1.2 review 时才补。

**轻微漏 L2**:spec §三 body 颜色规范不明确,导致方案选择时缺锚点。

## 3. Later Exposure

### V0.1.2 暴露的副作用

- **dark mode 推迟**:当时尝试加 dark mode,发现 token 反转复杂,推到 V0.2.x
- **window inactive 数据驱动延迟**:`useWindowActive` hook 的 focus/blur 事件在 macOS 上有 ~50ms 延迟,导致切换窗口时 vibrance 变化不丝滑。接受作为已知 trade-off

### V0.1.3 / V0.1.4 才暴露

- text-text-3 在 glass-l2/l3 上对比度 < 3:1(V0.1.3 修了)
- input/textarea focus 环弱 + cursor 行为不全(V0.1.3 修了)
- 玻璃阴影偏蓝(V0.1.3 修了 — shadow tint bug)
- V0.1.1 漏 L3 副作用(text select / resize / cursor)— V0.1.4 修了

## 4. Generalized Lesson

> **修视觉问题严格按 spec 数字走,不靠肉眼**。spec 缺口先补 spec,再修实现 — 不在 issue 里绕过。

未来遇到视觉修复:
- [ ] 每个改动都引用 spec §X.Y 章节
- [ ] spec 缺口先开 issue 补 spec,再修实现
- [ ] 视觉对比用工具(像素 diff / axe-core),不靠"看起来对"

## 沉淀

- 没新增 memory(V0.1.2 主要走了正确流程,没有可推广的"教训"层级)
- 但 V0.1.2 验证了"spec 主动恢复 + Layer 3 主动枚举"是正确模式,可作为正面例子引用
