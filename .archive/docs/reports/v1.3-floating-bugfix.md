## 🐛 Bug 现象

用户在浮动窗输入任务内容后点击"⏵ 开始计时"按钮：
- ✅ FOCUS 标签出现
- ✅ 任务标题出现
- ❌ **计时不开始**（时间显示 NaN 或 0，不递增）
- ✅ 暂停 / 继续 / 完成 / 撤销 按钮可点（点击事件能触发后端 command）

---

## 🔍 根因

**TypeScript 类型与 Rust 运行时之间的"类型谎言"导致前端拿到不完整数据。**

### 完整证据链

| 层 | 真相 |
|---|---|
| ① 前端点击 | `ExpandState.tsx:97-102` → `submit()` → `api.taskCreate()` + `api.taskStartTimer()`（line 20-21）— **正确** |
| ② 后端 `task_start_timer` | `src-tauri/src/db/task.rs:66-98` 正确设 `focus_started_at = now` + `focused_count + 1` — **9 个状态机测试全过** |
| ③ 事件 | `emit("focus-changed", &t)` 发出**完整 Task** — **正确** |
| ④ ❌ **Hook 调错** | `useActiveTask.ts:16` 调 `api.recordGetActive()` 而非 `api.recordGetActiveTask()` |
| ⑤ 后端 `record_get_active` | `db::record::get_active_task` SELECT **6 列**（缺 `duration_ms` / `focus_started_at`） |
| ⑥ TS 类型谎言 | `tauri-bridge.ts:20-29` `Record` 类型**谎报**有 `duration_ms` / `focus_started_at`（后端从未返回）—— **tsc 不报错** |
| ⑦ FocusBlock 计算 | `active.duration_ms + (active.status === "active" && active.focus_started_at ? now - active.focus_started_at : 0)` → `undefined + 0 = NaN` → `formatDuration(NaN)` |

**对比正常路径**：`record_get_active_task` command（`commands/record_cmd.rs:41-48`）通过 `record::get_active_task` + `task::get_task(source_id)` 拼出完整 12 字段 Task — 但 hook 从未调用它。

### Phase 2-4 摘要

- **Phase 2（模式）**：`recordGetActiveTask` 在 tauri-bridge 和后端都存在，是计划 Task 20 原本就为修此问题设计的方法 — hook 选错了入口
- **Phase 3（假设）**：`useActiveTask` 改用 `recordGetActiveTask` + 返回类型改 `Task | null`，下游 `active.source_id` → `active.id`
- **Phase 4（修复）**：5 个文件，最小改动 +44 / -15 行

### 回归测试覆盖

`active_task_view_links_to_full_task_with_timing_data`（位于 `db::record::tests`）：

```
1) start_timer 后 get_active_task 必须找到 active record
2) 那个 record 的 source_id 必须能查到完整 Task
3) 完整 Task 必有 focus_started_at: Some(...)
```

任何后续重构如果断了 `start_timer` / `get_active_task` / `get_task` 三者中的任一环，这个测试会立即失败。
