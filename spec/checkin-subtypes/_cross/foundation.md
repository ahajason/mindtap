# 打卡子场景共享基础 · 业务规格

> 7 个 check_in 子场景(情绪 / 作息 / 饮水 / 服药 / 习惯 / 饮食 / 复盘)共享的数据模型、浮窗入口与统一记录视图。

## §1 功能需求（F）

- **F-1** 用户在浮窗 StatusDot 点击展开后,可从已启用的子场景中选择一个快速打卡
  - R-1.1 子场景列表按用户启用顺序排列,已打卡(今日)的子场景以半透明样式显示
  - R-1.2 用户未启用任何子场景时,浮窗提示"请到主窗启用子场景"
- **F-2** 每条子场景打卡自动出现在主窗"全部记录"列表中
  - R-2.1 列表按提交时间倒序排列
  - R-2.2 单条记录显示子场景标签 + 摘要 + 时间
- **F-3** 用户可在主窗按子场景过滤"全部记录"列表
  - R-3.1 多选过滤(同时显示多个子场景)
  - R-3.2 顶部 chip 显示当前过滤条件 + 清除按钮
- **F-4** 用户可在主窗设置中启用 / 禁用任意子场景
  - R-4.1 默认启用:情绪 / 习惯 / 饮水
  - R-4.2 禁用子场景后,其在浮窗不可见,但历史记录保留
  - R-4.3 重新启用后历史记录立即可见

## §2 业务规则（R）

- **R-0.1** 7 个子场景共享同一基础记录结构:每条打卡 = check_in 记录 + 同步双写到统一记录视图
- **R-0.2** 浮窗 StatusDot 仅显示当前活动的 task 状态,子场景列表通过点击展开进入
- **R-0.3** 跨子场景搜索走主窗顶部 search bar,不在浮窗提供
- **R-0.4** 数据导出按"全部 check_in"范围,用户可选包含 note

## §3 业务边界（G / NG）

### Goal

- **G-1** 提供 7 个子场景共用的入口、记录视图、过滤、导出能力,避免每个场景重复造轮子
- **G-2** 维持"1 个浮窗入口 + 1 个主窗视图"的双窗架构,子场景作为场景化扩展

### NonGoal

- **NG-1** 本文件不涉及"7 个子场景各自的专属功能 / 数据 / 触发"——见 `../emotion.md` 等各 scene 文件
- **NG-2** 本文件不涉及"子场景的隐私 / 时区 / 通知策略"——见 `privacy.md` / `timezone.md` / `notifications.md`
